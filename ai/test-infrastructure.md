# Integration test infrastructure: `code"""..."""` DSL + TestWorkspace + TestClient

## Context

The current test suite has two tiers: pure in-memory unit tests (symbol/trie/bloom) and frozen
pre-compiled fixture JARs. `architecture-improvements.md:330` flags this as a stopgap. The
question is how to build toward three things: a `code"""..."""` marker DSL, a `TestWorkspace`
compiler harness, and a `TestClient` that simulates editor behaviour for full-stack LSP tests.

---

## Three-layer test architecture

```
Layer 3 (full-stack LSP)    TestClient — drives ServerImpl directly, captures notifications
Layer 2 (index integration) TestWorkspace — compiles .scala sources, returns live IndexManager
Layer 1 (unit)              Existing specs — in-memory symbols, ASM-generated bytecode
```

Each layer reuses the one below. `TestClient` uses `TestWorkspace` for compilation.
The `code"""..."""` DSL is useful at all three layers for expressing positions.

---

## Proposal 1 — Committed workspace directory ("clone into tmp")

**Assessment:** wrong tool for index tests; right tool for full-stack BSP tests that don't exist
yet. A committed Mill project adds maintenance burden and CI weight (Mill startup + BSP handshake)
without buying anything that `TestWorkspace` (Layer 2) doesn't already provide more cheaply.

**Recommendation:** defer until Phase 3+ when testing BSP negotiation or file-watching. Use
`TestWorkspace` instead for everything index-related.

---

## Proposal 2 — `code"""..."""` string interpolator

Strips marker tokens from source text and records their line/column. No compiler dependency.

```scala
import util.Code.*

val src = code"""
  |object Foo {
  |  val x: ${m1}Int${m2} = 42
  |}
"""
// src.text   → source text without markers
// src.at(m1) → lsp.Position(line=2, char=9)
// src.at(m2) → lsp.Position(line=2, char=12)
```

Implementation is ~80 lines, almost directly ported from dotty's `Code.scala` /
`CodeMarker` / `textAndPositions`. Key differences:
- Weaver/IO instead of JUnit — keep it a plain data type, no implicit `→ CodeTester` conversions
- Positions are `lsp.Position` (already on classpath via smithy4s-generated types)
- No `TastyWithPositions` variant initially

**Files:**
- `sls/test/src/…/util/CodeMarker.scala` — `case class CodeMarker(name: String)`
- `sls/test/src/…/util/Code.scala` — extension `code"""..."""`, `SourceWithPositions(text, positions)`

## Proposal 3 — `TestWorkspace` (Layer 2, index integration)

Compiles source strings with in-process dotc, returns a live `IndexManager` / `SymbolIndex`.

```scala
TestWorkspace.withSources(
  "Foo.scala" -> src.text,
  "Bar.scala" -> bar.text,
).use { ws =>
  for {
    idx <- ws.symbolIndex
    hits = idx.atPosition(ws.uri("Foo.scala"), src.at(m1))
  } yield expect(hits.exists(_.id == SymbolId("example/Foo#x.")))
}
```

**Compilation strategy:** `dotty.tools.dotc.Main` in-process (same as dotty's `TestServer`).
- The Scala 3 compiler must be on the test runtime classpath — one `ivyDeps` line in `build.mill`
  for `org.scala-lang::scala3-compiler:3.8.3` scoped to `sls.test`
- Output goes to `os.temp.dir(prefix = "sls-test-ws-")` (matches existing pattern)
- `IndexManager` is pointed at the classes dir and initialised by running the TASTY/bytecode
  indexers over the output

**Surface:**
```scala
case class TestWorkspace(
  sourceRoot: os.Path,
  classesRoot: os.Path,
  sources: Map[String, os.Path],
) {
  def uri(name: String): SourceUri
  def symbolIndex: IO[SymbolIndex]
  def indexManager: IO[IndexManager]
}

object TestWorkspace {
  def withSources(srcs: (String, String)*): Resource[IO, TestWorkspace]
}
```

**Files:**
- `sls/test/src/…/util/TestWorkspace.scala` — ~100 lines

## Proposal 4 — `TestClient` (Layer 3, full-stack LSP)

Implements `SlsLanguageClient[IO]` (the smithy4s-generated interface that `ServerImpl` pushes
notifications to) and holds a reference to the server under test. Helper methods simulate the
burst of LSP requests a real editor fires.

### What a real editor sends on file edit

VS Code sends incremental `textDocument/didChange` per keystroke — not a single full-document
replace. `TestClient.editFile` must model this:

```scala
// Simulates typing "List" one character at a time, triggering didChange for each
client.editFile(uri, position, "List")
// internally: L → didChange, Li → didChange, Lis → didChange, List → didChange
```

Or at minimum, a multi-step replacement that mirrors VS Code's actual protocol (delete + insert
as two separate content changes in one `didChange`, rather than one big replace).

### Interface

```scala
class TestClient(server: SlsLanguageServer[IO]) extends SlsLanguageClient[IO] {

  // --- SlsLanguageClient[IO] impl (captures server notifications) ---
  private val diagsRef: Ref[IO, Map[String, List[lsp.Diagnostic]]] = ...

  override def textDocumentPublishDiagnostics(params: PublishDiagnosticsParams): IO[Unit] =
    diagsRef.update(_.updated(params.uri, params.diagnostics.toList))

  override def windowLogMessage(params: LogMessageParams): IO[Unit] = IO.unit
  override def windowShowMessage(params: ShowMessageParams): IO[Unit] = IO.unit
  override def initialized(params: InitializedParams): IO[Unit] = IO.unit

  // --- Editor simulation helpers ---

  /** Open a file exactly as VS Code does (didOpen). */
  def openFile(uri: String, text: String): IO[Unit]

  /** Send one or more incremental didChange notifications, one per provided ContentChange. */
  def changeFile(uri: String, changes: lsp.TextDocumentContentChangeEvent*): IO[Unit]

  /**
   * High-level helper: replaces the range [start, end) with `newText`, emitted as a
   * single-change didChange (the common VS Code pattern for autocomplete insertion).
   */
  def editFile(uri: String, range: lsp.Range, newText: String): IO[Unit]

  def saveFile(uri: String): IO[Unit]
  def closeFile(uri: String): IO[Unit]

  /** Get the latest diagnostics published for `uri`. */
  def latestDiagnostics(uri: String): IO[List[lsp.Diagnostic]]
}
```

### Construction — stubbing ServerImpl's collaborators

`ServerImpl` has many constructor parameters (pcProvider, bspStateManager, cspClientDeferred,
etc.). For full-stack tests that focus on LSP-level behaviour (document sync, diagnostics) we
need test doubles. Pattern:
- Keep the real `TextDocumentSyncManager` and `DiagnosticManager`
- Stub `BspStateManager`, `CspServer`, `PresentationCompilerProvider` with `IO.never` / no-op
  implementations
- Provide `Deferred.of[IO, BuildServer].map(_.complete(stub))` for `bspClientDeferred`
- `TestComputationQueue` already exists in `DocumentSyncTests` — reuse it

```scala
object TestClient {
  /** Build a TestClient + ServerImpl wired together, with all heavy dependencies stubbed. */
  def resource(workspace: TestWorkspace): Resource[IO, TestClient]
}
```

This means `TestClient.resource(ws)` is the single entry point for Layer 3 tests.

**Files:**
- `sls/test/src/…/util/TestClient.scala` — ~150 lines
- Extract `TestComputationQueue` from `DocumentSyncTests` into `sls/test/src/…/util/`

## Combined usage example

```scala
test("diagnostics appear after a file edit introduces a type error") {
  val src = code"""
    |object Foo {
    |  val x: ${m1}String${m2} = 42
    |}
  """

  TestWorkspace.withSources("Foo.scala" -> src.text).flatMap { ws =>
    TestClient.resource(ws)
  }.use { client =>
    for {
      _    <- client.openFile(ws.uri("Foo.scala"), src.text)
      diag <- client.latestDiagnostics(ws.uri("Foo.scala"))
    } yield expect(diag.exists(d => d.range.start == src.at(m1)))
  }
}
```

## Phasing

**Now (index phase, implement immediately):**
1. `CodeMarker` + `code"""..."""` — no compiler dependency, trivially useful
2. `TestWorkspace` with in-process dotc — unblocks real TASTY-driven index tests

**Later (full-stack LSP phase):**
3. `TestClient` + stub collaborators — when testing didChange → diagnostics round-trips
4. Committed workspace directory fixture — when testing BSP negotiation or file watching

---

## Critical assessment of `code"""..."""` vs hardcoded positions

Dotty's pattern shines when you have many tests at many positions in the same snippet. For index
tests (where the query is usually "does this symbol appear anywhere in this file?") hardcoded
positions are already rare. The DSL pays off most in Layer 3 (hover, completion, go-to-definition)
where you need to say "query at *this exact character* in this code". Building it now is fine
because it's tiny; just don't force it onto tests that don't need it.
